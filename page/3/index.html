原理

下面来阐述一下 let 从句在内存里操作的原理。

let 从句的赋值行为并非实际上写法这样，也就是说，它并不是一种直接的赋值行为。在内存里，它会创建出一个匿名类型（Anonymous Type）。在 C# 的设计里，匿名类型使用的是创建一个类的模式来搞定，只是这个类的类名我们不关心，于是这个类就让 C# 的编译器来创建了，而不需要我们创建。

当我们声明了 let Capital = word[0] 的时候，系统就会为我们创建一个匿名类型。这个类型使用 IDE 来查看，你将看到它叫做 'a 类（把鼠标放到 Capital 上就会看到这个 'a 类）。在这个所谓的 'a 里，它包含一个属性，这个属性是 Capital。

可以看到，匿名类型会自动创建的成员是属性，而并非字段。于是我们使用的时候，实际上应当按照命名格式，为这个使用 let 从句创建出来的所谓的临时变量用 PascalCase（帕斯卡）命名法，而不是 camelCase（驼峰）命名法。

于是，我们就可以通过整个语句环境下，可以任意操作整个 'a 匿名类型。

方法链写法

下面提一下方法链的写法。

我们来仔细分析一下它的实现原理。在上文之中，我们已经知道 Select 方法和 Where 方法的具体实现模式和逻辑。大致分为两个特点：

  1. 推迟执行（Deferring Execution）特点：Where 不会立马执行筛选，而是推后给出一个新的对象 WhereEnumerableIterator<T> 对象，而 Select 作为语句末端，才会执行。
  2. 方法链调用顺序是类似于 A.B().C().D() 的方式，那么可以暗示 A.B() 得到的类型可以使用 C 方法，而执行 C 方法后得到的类型，可以调用 D 类型。

值元组和匿名类型的对比

还需要注意的是，C# 7 开始支持值元组（Value Tuple）类型，这种类型是 C# 4 提供的元组（Tuple）的轻量级版本。它采用值类型结构完成实现，使得生成这些类型都会相较于类快一些，而且支持任意时刻命名每一个创建好的字段。

这就需要你注意两点内容了。值元组和匿名类型有非常相似的地方：它们最终都是操作内部的内容，并且它们的名称（类名和结构名）我们并不关心，这些都会交给编译器生成。但需要注意的地方是，匿名类型创建出来的成员是属性，而值元组创建出来的却是字段。这是我们第一个需要注意的地方。

第二个需要注意的是，由于 LINQ 机制的问题，很遗憾的是，到 C# 8 都依然不支持在 let 语句里使用值元组的模式来创建更快捷更轻量级的值元组的模式。这并非意味着我们无法在 LINQ 里面使用值元组。这仅仅是想说明，值元组并不能用于 let 从句里的解构（即用于赋值语句的左侧，即左值表达式）。

返回完成的任务

问题

如何实现一个具有异步签名的同步方法。如果从异步接口或基类继承代码，但希望用同步的方法来实现它，就会出现这种情况。对异步代码做单元测试，以及用简单的生成方法存根（stub）或者模拟对象（mock）来产生异步接口，这两种情况下都可使用这种技术。

解决方案

可以使用 Task.FromResult 方法创建并返回一个新的 Task<T> 对象，这个 Task 对象是已经完成的，并由具体的值。

如果你使用了 Microsoft.Bcl.Async，那么 FromResult 方法就在 TaskEx 类之中。

讨论

在用同步代码实现异步接口的时候，要避免使用任何形式的阻塞操作。在异步方法之中进行阻塞操作，然后返回一个完成的 Task 对象，这种方法并不可取。作为一个反例，我们来看一下 .NET Framework 4.5 之中 Console 类的文本读取器。Console.In.ReadLineAsync 一定会阻塞调用它的线程，直到它读取完一行文字，然后回返回一个已经完成的 Task 对象。这种实现并不直观，很多开发人员也觉得很奇怪。一旦异步方法阻塞，调用它的线程就无法启动其它任务，这会干扰程序的并发性，甚至可能产生死锁。

Task.FromResult 只能提供结果正确的同步 Task 对象。如果要让返回的 Task 对象有一个其它类型的结果（例如以 NotImplementedException 结束的 Task 对象），就得自行创建使用 TaskCompletionSource 的辅助方法。

运算符是什么？

运算符是程序执行操作期间，为了避免复杂的函数调用和书写格式，通过一些简单的、类似于数学表达式的模式书写的语言规范。在 C 语言里，我们可以尝试使用加号来拼接两个数字，得到两个数字的和。

其中的 = 叫做赋值运算符，它被称为一个广义的运算符，因为它实际上并不称为一种运算，而是一种行为：从微观角度来看，它是把符号右侧的数据结果，放到变量 i 存放的位置上；从宏观来看，其实就是使得 i 得到 5 的这个结果的行为，就称为赋值（Assignment）。

现在考虑到这个的时候，a 和 b 的求和结果肯定就是 float 类型所表示的 3.5 了。但你仔细观察代码，就可以发现，result1 可是 int 类型的，这意味着这个它想把 float 的结果用 int 表示出来，所以最终 3.5 被程序截断（注意，并不是四舍五入），于是这个数就变为了 3；同理，第二个则完全不需要转换，所以 result2 一定是 3.5 这个结果。

那么，这里就学习到了一个知识点，叫做类型转换（Type Conversion）。类型转换分两种，一种是人为强制定义这个结果转换为另一种类型的模式，叫做强制类型转换或显式转换（Explicit Conversion）；而另外一种，就是上面这样，通过系统自己的机制进行的类型转换，叫做隐式转换（Implicit Conversion）。

但是，可以发现，不论类型较小转较大，还是较大转较小，C 语言貌似都给出了隐式转换的模式和手段。比如 float 转 int 就是典型的大转小，而它采用的手段是截断（Truncate），而不是四舍五入（Round），所以在使用这种转换的时候应小心数据类型不同时引发的一些潜在的问题。

IDisposable 接口简述

在 C# 里，我们是允许使用不安全代码（指针）来操作非托管内存的，于是便产生了非托管内存如何规范释放的问题。IDisposable 接口的产生，正是为了这种需求。

当我们需要手动释放一些比较大的资源的时候（比如 GC 没有回收垃圾资源，或是由于 GC 回收机制体系的 bug 导致无法回收的时候，我们就需要这个接口了。

IEnumerable<T> 类型简介

这个类型是一个泛型类型，它表示一个可以使用 foreach 循环的基本列表接口类型。只要你实现了 IEnumerable<T> 里那一套成员，你就可以把这个类型的所有数据都用 foreach 循环遍历一遍。所以这个类型被称为“可枚举类型”、“可迭代类型”、“可列举类型”等等。其中的 T 是泛型参数，表示一个实体化类型，但目前还没有实现。当我们使用这个类型的时候，需要提供这个类型参数，并最终生成一个与该类型配套的完整的类型。比如前文里的 IEnumerable<int> 就是表示为元素为 int 类型的可枚举序列。

LINQ 可以使用查询表达式，也可以使用方法链来书写，这取决于你的风格和思维方式。如果你绝对查询表达式写起来好看，也喜欢写这种的话，就写这种；如果你觉得方法链更利于学习原理的话，你可以使用方法链。这两种写法在运行期间是等价的，也就是说，不论是哪一种写法，最终都会被翻译为完全一样的代码。

方法链模式

不过，LINQ 还有方法链的写法。我们也来看一下。

那么，我们来阐述一下，这个方法链到底是怎么起作用的。

首先，在 LINQ 的内部实现之中，Where 方法计算的表达式会被推后执行，因为执行这样的语句是耗时间的，程序为了避免立马执行，就先考虑到性能的问题，把条件信息和元素的数据信息全部提取出来，放在一旁，产生出一个可执行筛选的对象（实际上称为 WhereEnumerableIterator<T> 对象）。这个对象并没有立马得到执行，它此时只是一个提供了筛选功能的对象而已。而真正需要执行筛选的操作，则直接推后到了 foreach 循环调用的时候，不得不去遍历该集合时，内部的元素才会得到遍历，并执行筛选。

所以，可以看出来，微软的工程师们还是很厉害的，想到了一个万全之计推后了 foreach 的执行，同时也解决了影响性能的问题。当这个集合一直没有被 foreach 调用到，那么这个语句就只是一个语句，生成的对象也不会得到任何结果。比如，上述的四个变量都是这么去书写的，导致得到的都是这个所谓的 WhereEnumerableIterator<T> 对象，而这个对象在没有得到 foreach 调用的情况下，我们如果此时尝试不执行 foreach 循环的前提下就取出元素，那么这样是否可能呢？是可能的，但这样做只会让程序陷入“危险”的境地，因为元素并没有没迭代，就想要获取里面的元素，这显然是不现实的。

有没有 Select(d => d) 很重要吗？

回到前文给出的代码，我们发现最后有一个 Select(d => d) 这样写法的方法。我们知道，Select 方法执行的是一种映射关系，在 foreach 循环的每一次迭代过程上，临时变量 d 都会被映射到 Lambda 表达式后给出的结果上来。但实际上，这样写貌似一点映射也没变，对每一个元素 d 进行迭代，然后返回的依然还是它自己。那写这个和不写这个有没有区别呢？

实际上是有的。不写的话，就不会执行 foreach 的迭代过程，也就意味着这个列表并没有在此时就开始执行，并得到结果。由于 LINQ 有推迟执行的特效，导致如果不写 Select 的话，执行就不会在这个时候开始，而是推后到你后面的某处 foreach 代码，或是调用聚合函数，才会触发 foreach 的调用。否则，这里就只是一个可迭代对象。

如果你想在这里就给出结果，请添加上 Select(d => d) 方法；而如果后续你一定能保证会执行 foreach 循环，那么就不用写了。
